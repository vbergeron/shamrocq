## Identified patterns

### 1. GLOBAL → LOAD → CALL → SLIDE: partially-applied global call

The single biggest pattern in large tests.  A 2-arg global call like
`(eqb ord x)` compiles to:

```
GLOBAL  8 (eqb)      ; push closure from global table
LOAD    1             ; push first arg
CALL                  ; → enters eqb/0: LOAD 0; CLOSURE <eqb/0/0> cap=1; RET
                      ;   allocates a closure on the heap
LOAD    3             ; push partial-apply result
CALL                  ; call the partial-apply closure with second arg
SLIDE   1             ; drop intermediate
```

**Cost**: 2 CALL dispatches, 1 heap-allocated closure, 1 SLIDE — for what is
logically a single 2-arg call.

In hash_forest: 83 `GLOBAL;LOAD` pairs, 90 `CALL;SLIDE` pairs.  The 14
occurrences of `eqb` are all partial-apply sites.

### 2. CLOSURE;RET curried stubs (~10% of bytecode)

74–100% of all CLOSURE instructions are in stub bodies that do nothing but
capture one argument and return a new closure:

```
<add/0>: ; bare_fn
  LOAD    0           ; arg
  CLOSURE <add/0/0> captures=1
  RET
```

| Test | Stubs / Closures | Stub bytes | Code bytes | Overhead |
|------|-----------------|------------|------------|----------|
| synth_arith | 10 / 10 (100%) | 50 | 464 | 11% |
| synth_hof | 13 / 13 (100%) | 65 | 213 | 31% |
| synth_option | 9 / 9 (100%) | 45 | 237 | 19% |
| hash_forest | 81 / 109 (74%) | 405 | 4284 | 9% |

### 3. /direct body duplication (~30% of bytecode)

The known-arity direct call optimization emits a separate "flat" body for each
multi-arity function.  This duplicates the logic of the curried body:

| Test | /direct bytes | Code bytes | Duplication |
|------|-------------|------------|-------------|
| synth_arith | 154 | 464 | 33% |
| synth_list | 113 | 406 | 28% |
| hash_forest | 1386 | 4284 | 32% |
| synth_tree | 1959 | 5779 | 34% |

Both bodies are needed today: the curried body for partial application, the
direct body for fully-saturated calls.

### 4. Single-case MATCH (unnecessary dispatch)

Many `MATCH` instructions have exactly 1 case — a guaranteed-to-succeed
destructure:

```
MATCH  1 cases:
  | tag=11 (Build_hforest) arity=2 -> 0322
BIND   2
```

| Test | 1-case MATCHes | Total MATCHes | Ratio |
|------|---------------|---------------|-------|
| hash_forest | 62 | 112 | 55% |
| synth_tree | 62 | 155 | 40% |
| synth_arith | 0 | 16 | 0% |

Each 1-case MATCH costs 6 bytes (opcode + n_cases + tag + arity + offset) for
a branch that always succeeds.

### 5. LOAD;LOAD chains

`LOAD;LOAD` is the #1 bigram (196 in hash_forest).  Each LOAD is a 2-byte
instruction that pushes one value from the frame.  These almost always set up
arguments for a subsequent CALL or CTOR.

### 6. Indirect vs direct call balance

In hash_forest, many globals are still called indirectly despite having known
arity.  The compiler only emits CALL_DIRECT when **all** arguments are provided
at the call site; partial application falls back to indirect calls:

| Global | Indirect calls | Direct calls | Arity |
|--------|---------------|-------------|-------|
| eqb | 14 | 6 | 3 |
| map | 12 | 2 | 2 |
| filter | 12 | 4 | 2 |
| existsb | 10 | 2 | 2 |
| negb | 8 | 0 | 1 |

## Runtime memory profile

### Baseline: `hforest_lifecycle`

| Metric | Value |
|--------|-------|
| Peak heap | 2952 B |
| Peak stack | 412 B |
| Closure allocations | 215 |
| Ctor allocations | 92 |
| Total alloc bytes | 2880 B |
| Indirect CALL | 414 |
| TAIL_CALL | 58 |
| CALL_DIRECT | 119 |
| TAIL_CALL_DIRECT | 144 |
| Instructions | 5943 |
| Peak call depth | 9 |

Closures dominate the heap: ~2150 B out of 2880 B (75%).  Since the heap is
bump-only (no GC), every intermediate closure persists for the entire
execution.

### Closure allocation breakdown (bytecode-level)

In hash_forest bytecode, 109 CLOSURE instructions split into:

| Category | Count | Heap cost each | Notes |
|----------|-------|---------------|-------|
| bare_fn (0 captures) | 45 | 0 B | Encoded in Value word |
| 1 capture (25 in stubs) | 31 | 8 B | header + 1 cap |
| 2 captures (21 in stubs) | 27 | 12 B | header + 2 caps |
| 3 captures (3 in stubs) | 5 | 16 B | header + 3 caps |
| 4 captures (1 in stubs) | 1 | 20 B | header + 4 caps |

Of the 64 heap-allocating CLOSURE instructions, **50 are in curried stubs**
(CLOSURE;RET bodies).  Each stub executes once per indirect call to its parent,
producing a closure that:
- For the **final** currying step: IS the returned value (unavoidable)
- For **intermediate** steps: is garbage the moment the next CALL consumes it

### Call stack overhead

Each indirect CALL pushes a `CallFrame { return_pc: usize, frame_base: usize }`
(16 bytes on 64-bit, 8 bytes on 32-bit) plus restores captures onto the
operand stack.  The capture-restore loop (`closure_capture(func, i)` for each
capture) is a heap read per capture value.

A curried stub call therefore costs:
1. Push CallFrame (call stack)
2. Read closure header from heap (code_addr + n_captures)
3. Copy N captures from heap to stack
4. Execute stub body (LOAD; CLOSURE; RET = 3 instructions)
5. Allocate new closure on heap
6. Pop CallFrame (return)

For a 3-arity function called with 2 args (e.g., `eqb ord x`), this happens
**twice**, producing one intermediate closure (8 B) that becomes garbage.

### Per-optimization heap impact

| Optimization | Heap savings per site | Sites (hash_forest) | Est. runtime savings |
|--------------|-----------------------|---------------------|---------------------|
| **UNPACK** | 0 B | 62 | — (CPU only) |
| **Partial-apply direct** | 8 B per intermediate closure | 14 (eqb, 3-arity × 2 args) | ~112 B |
| **Partial-apply direct** | 0 B (2-arity, result IS the closure) | 42 (map, filter, etc.) | — (CPU only) |
| **Deduplicate /direct** | 0 B (code blob, not heap) | — | — |
| **LOAD2** | 0 B | 196 | — |
| **CALL_SLIDE** | 0 B | 90 | — |

### Stack depth impact

| Optimization | Stack effect |
|--------------|-------------|
| **Partial-apply direct** | −1 call depth per skipped stub (peak 9 → ~7) |
| **CALL_SLIDE** | Fused drop reduces peak operand stack by N words |
| **UNPACK** | No change (same BIND effect) |
| **LOAD2** | No change (same values pushed) |

### Where heap savings matter most

On the embedded target (30–64 KB arena), the bump-only allocator means every
byte of garbage closure persists until execution ends.  The 215 closures in
`hforest_lifecycle` consume ~2150 B — **73% of the 2952 B heap**.  The
intermediate closures from curried stubs are the clearest waste: they're used
exactly once (by the next CALL) and immediately become unreachable.

For larger inputs (longer lists, deeper trees), the closure count scales with
the input size while ctor allocations stay proportional to output.  A function
like `map f xs` allocates **one stub closure per element** (plus the output
cons cells).  Eliminating those stubs via direct partial-apply or native
multi-arity would make closure allocation O(1) instead of O(n) for such
combinators.

## Optimization opportunities

### A. UNPACK opcode (replaces 1-case MATCH)

Replace `MATCH 1 case; BIND N` with `UNPACK arity` (2 bytes).  Directly binds
the constructor's fields without branching or tag comparison.

- **Saves**: 62 × 4 = 248 bytes in hash_forest
- **CPU**: eliminates 62 branch dispatches per execution
- **Complexity**: low — new opcode, compiler pattern match

### B. Deduplicate /direct bodies

Instead of emitting two complete copies, emit a single canonical body usable
by both the curried and direct entry paths.  The direct entry point could
rearrange the stack frame to match the curried layout, or the body could be
parameterized by frame layout.

- **Saves**: ~30% code size across all tests
- **Complexity**: medium — compiler/VM calling convention change

### C. LOAD2 fused opcode

`LOAD2 a b` (3 bytes) replaces `LOAD a; LOAD b` (4 bytes).  Reduces dispatch
count and code size for the most common bigram.

- **Saves**: ~196 bytes + 196 fewer dispatches in hash_forest
- **Complexity**: low — new opcode

### D. CALL_SLIDE fused opcode

`CALL_SLIDE n` (2 bytes) replaces `CALL; SLIDE n` (3 bytes).  Fuses the most
common post-call cleanup.

- **Saves**: 90 bytes + 90 fewer dispatches in hash_forest
- **Complexity**: low — new opcode

### E. Partial-application direct call

`CALL_DIRECT` today only fires for **fully-saturated** calls (all N args
provided).  The data shows many globals are still called indirectly because
they are partially applied at the call site:

| Global | Arity | Indirect calls | Direct calls |
|--------|-------|---------------|-------------|
| eqb | 3 | 14 | 6 |
| map | 2 | 12 | 2 |
| filter | 2 | 12 | 4 |
| existsb | 2 | 10 | 2 |
| negb | 1 | 8 | 0 |

A partially-applied 2-arg call like `(map f)` currently compiles to:

```
GLOBAL  3 (map)       ; push bare-fn closure
LOAD    x             ; push arg
CALL                  ; enters map/0 stub → LOAD 0; CLOSURE <map/0/0> cap=1; RET
                      ; allocates a closure on the heap
```

An extended `CALL_DIRECT <map/0/0> args=1` could jump directly to the inner
curried body, skipping the stub dispatch.  For 3-arity functions called with
2 args (like `eqb ord x`), this avoids **one intermediate closure allocation**
and **one full CALL dispatch** through the stub chain.

- **Saves**: 83 stub dispatches + many intermediate closures in hash_forest
- **CPU**: eliminates stub entry/exit overhead per partial-apply site
- **Complexity**: medium — compiler must resolve partial-apply targets and
  emit CALL_DIRECT with under-saturated arity
